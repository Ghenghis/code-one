# Tier 4.5 — Automation Fabric (Agentic Upgrade)

Date: 2026-04-14

## Objective

Introduce a continuous automation layer that operates between every task to maximize agent autonomy, verification, efficiency, and system intelligence.

---

## Tier Definition

### Tier 4.5 — Automation Fabric

This layer wraps the Agent System (Tier 4) and provides:
- continuous supervision
- autonomous validation
- dynamic routing
- persistent improvement

---

## Core Components

### 1. Shadow Verifier
- Secondary model reviews all outputs before completion
- Checks: tests, diff quality, architecture alignment, missing artifacts

### 2. Model Router
- Routes tasks dynamically:
  - reasoning model
  - editing model
  - utility model
  - vision/browser model

### 3. Approval Broker
- Smart approval system:
  - auto-approve safe actions
  - escalate risky actions
  - batch approvals

### 4. Cost Governor
- Tracks token/API usage
- Switches models dynamically
- Enforces budget limits

### 5. Context Curator
- Updates:
  - repo map
  - architecture map
  - dependency map
  - runtime map
  - agent map
- Maintains compressed context

### 6. Eval Engine
- Continuous evaluation of:
  - patch success
  - task completion
  - agent performance
- Stores metrics

### 7. Failure Recovery Daemon
- Checkpoints all tasks
- Resume from failure
- Rollback bad changes

### 8. Skill Trigger Engine
- Auto-runs skills based on context:
  - deployment -> deploy validator
  - CI -> CI checker
  - UI -> visual test agent

### 9. Computer Use Layer (Optional)
- UI automation
- Web dashboard interaction
- End-to-end validation

## Provider Strategy

### Role-to-Provider Mapping

| Role | Primary | Fallback 1 | Fallback 2 | Fallback 3 |
|---|---|---|---|---|
| Reasoning/Planning | Claude Opus 4.6 | MiniMax-M1 | GPT-4o / o3 | Local (DMax-Coder-16B) |
| Editing/Code Gen | Claude Sonnet 4.6 | MiniMax-M1 | GPT-4o-mini | Local model |
| Utility (summaries, commits) | Claude Haiku 4.5 | GPT-4o-mini | MiniMax | Local model |
| Vision/Browser | Claude Opus 4.6 | GPT-4o | Gemini | - |
| Shadow Verification | MiniMax-M1 | Claude Sonnet 4.6 | GPT-4o | - |
| Embeddings | Local model | OpenAI embeddings | - | - |

### Fallback Rules

- Every model role has an ordered provider chain
- On rate limit (429): retry with backoff, then fall to next provider
- On server error (5xx): immediately fall to next provider
- On timeout: fall to next provider
- On auth failure: skip provider, mark unhealthy, alert user
- Health checks run periodically to re-enable recovered providers
- All provider switches logged as ProviderFallbackEvents
- User can pin a specific provider to override automatic routing
- Cost governor can force downgrade to cheaper provider when budget is tight
- Local models serve as final offline fallback for all roles

### Seamless Switching

Provider switching is invisible to the agent loop. The AI Gateway:
- Normalizes all provider APIs to a single streaming interface
- Handles auth, request mapping, and response normalization
- Tracks token usage and cost per provider
- Reports provider health metrics to the Eval Engine
